Event Sourcing

Persistir el estado de una entidad como una secuencia inmutable de eventos en lugar de guardar solo el estado actual.

Arquitectura

📖 Descripción

En Event Sourcing, cada cambio de estado se registra como un evento inmutable en un log (event store). El estado actual se reconstruye reproduciendo todos los eventos. Provee historial completo de auditoría, permite reconstruir el estado en cualquier punto del tiempo y facilita la integración con CQRS y el debug en producción.

🎯 Casos de uso prácticos

1. Cuenta bancaria

El saldo se calcula sumando los eventos DepositoRealizado y RetiroRealizado. El historial completo está disponible para auditoría.

2. Carrito de compra

Los eventos ItemAgregado, ItemEliminado y CuponAplicado reconstruyen el estado actual del carrito en cualquier momento.

💻 Ejemplos de código

JavaScript
class EventStore {
  constructor() { this.eventos = []; }
  guardar(evento) {
    const registrado = { ...evento, timestamp: new Date().toISOString(), version: this.eventos.length + 1 };
    this.eventos.push(registrado);
    return registrado;
  }
  obtenerPara(agregadoId) { return this.eventos.filter(e => e.agregadoId === agregadoId); }
}

class CuentaBancaria {
  constructor(id) { this.id = id; this.saldo = 0; this.titular = null; }

  // Reconstruir estado desde eventos
  static desde(eventos) {
    const cuenta = new CuentaBancaria(eventos[0]?.agregadoId);
    eventos.forEach(e => cuenta.aplicar(e));
    return cuenta;
  }

  aplicar(evento) {
    switch (evento.tipo) {
      case 'CuentaAbierta':   this.titular = evento.datos.titular; this.saldo = 0; break;
      case 'DepositoRealizado': this.saldo += evento.datos.monto; break;
      case 'RetiroRealizado':   this.saldo -= evento.datos.monto; break;
    }
  }
}

// Uso
const store = new EventStore();
const cuentaId = 'C-001';

store.guardar({ agregadoId: cuentaId, tipo: 'CuentaAbierta',    datos: { titular: 'Cesar' } });
store.guardar({ agregadoId: cuentaId, tipo: 'DepositoRealizado', datos: { monto: 1000 } });
store.guardar({ agregadoId: cuentaId, tipo: 'DepositoRealizado', datos: { monto: 500  } });
store.guardar({ agregadoId: cuentaId, tipo: 'RetiroRealizado',   datos: { monto: 200  } });

const eventos = store.obtenerPara(cuentaId);
const cuenta  = CuentaBancaria.desde(eventos);
console.log(`Saldo de ${cuenta.titular}: €${cuenta.saldo}`); // €1300
console.log(`Historial: ${eventos.length} eventos`);
Java
public sealed interface Evento permits CuentaAbierta, Deposito, Retiro {
    String agregadoId();
}
public record CuentaAbierta(String agregadoId, String titular) implements Evento {}
public record Deposito(String agregadoId, double monto) implements Evento {}
public record Retiro(String agregadoId, double monto) implements Evento {}

public class CuentaBancaria {
    private String titular;
    private double saldo;

    public static CuentaBancaria desde(List<Evento> eventos) {
        CuentaBancaria c = new CuentaBancaria();
        eventos.forEach(c::aplicar);
        return c;
    }

    private void aplicar(Evento e) {
        switch (e) {
            case CuentaAbierta ev -> titular = ev.titular();
            case Deposito ev      -> saldo += ev.monto();
            case Retiro ev        -> saldo -= ev.monto();
        }
    }
    public double getSaldo() { return saldo; }
}

🚫 Cuándo NO usar

  • Cuando no necesitas historial de cambios ni auditoría; gestionar el event store es complejo.
  • Para entidades con estado extremadamente volátil (millones de eventos); requiere snapshotting.
  • Si el equipo no está familiarizado; la curva de aprendizaje y el overhead operacional son altos.
