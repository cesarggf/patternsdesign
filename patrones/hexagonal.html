Arquitectura Hexagonal

Aislar el núcleo de la aplicación (dominio + casos de uso) de los detalles tecnológicos externos, conectándolos mediante puertos e adaptadores.

Arquitectura

📖 Descripción

También conocida como Ports & Adapters (Alistair Cockburn), la arquitectura hexagonal coloca la lógica de negocio en el centro y define Puertos (interfaces) por donde se comunica con el exterior. Los Adaptadores implementan esos puertos: un adaptador HTTP, un adaptador de BD, un adaptador de email. El core no depende de ningún framework o tecnología concreta.

🎯 Casos de uso prácticos

1. Sistema de procesamiento de pagos

El core de pagos no sabe si usa Stripe o PayPal; el adaptador concreto se inyecta en runtime.

2. Servicio de notificaciones

El caso de uso 'notificar usuario' no conoce si el canal es email, SMS o push; lo decide el adaptador.

💻 Ejemplos de código

JavaScript
// Puerto (interfaz que el core define)
// En JS, los puertos son contratos implícitos / interfaces documentadas

// CORE — no depende de nada externo
class ServicioNotificaciones {
  constructor(canalNotificacion) {
    // Puerto inyectado (el adaptador concreto llega desde fuera)
    this.canal = canalNotificacion;
  }

  notificarPedidoConfirmado(usuario, pedido) {
    // Lógica de negocio pura
    const mensaje = {
      destinatario: usuario.email,
      asunto: `Pedido #${pedido.id} confirmado`,
      cuerpo: `Hola ${usuario.nombre}, tu pedido de €${pedido.total} fue confirmado.`
    };
    this.canal.enviar(mensaje);
  }
}

// ADAPTADORES (infraestructura)
class AdaptadorEmail {
  enviar({ destinatario, asunto, cuerpo }) {
    console.log(`[SMTP] → ${destinatario}`);
    console.log(`  Asunto: ${asunto}`);
    console.log(`  ${cuerpo}`);
  }
}

class AdaptadorConsola {
  enviar({ destinatario, asunto, cuerpo }) {
    console.log(`[Consola] Para: ${destinatario} | ${asunto}`);
  }
}

class AdaptadorSMS {
  enviar({ destinatario, cuerpo }) {
    console.log(`[SMS] → ${destinatario}: ${cuerpo.substring(0, 60)}...`);
  }
}

// Composición — el adaptador se inyecta según el entorno
const adaptador  = process.env.NODE_ENV === 'test'
  ? new AdaptadorConsola()
  : new AdaptadorEmail();

const servicio = new ServicioNotificaciones(adaptador);
servicio.notificarPedidoConfirmado(
  { nombre: 'Cesar', email: 'cesar@example.com' },
  { id: 42, total: 199.99 }
);
Java
// Puerto (interfaz del core)
public interface CanalNotificacion {
    void enviar(Mensaje mensaje);
}

// CORE — lógica de negocio pura
public class ServicioNotificaciones {
    private final CanalNotificacion canal;

    public ServicioNotificaciones(CanalNotificacion canal) { this.canal = canal; }

    public void notificarPedido(Usuario u, Pedido p) {
        canal.enviar(new Mensaje(
            u.getEmail(),
            "Pedido #" + p.getId() + " confirmado",
            "Hola " + u.getNombre() + ", pedido de €" + p.getTotal()
        ));
    }
}

// ADAPTADORES (infraestructura)
public class AdaptadorEmail implements CanalNotificacion {
    public void enviar(Mensaje m) {
        System.out.println("[SMTP] → " + m.destinatario() + " | " + m.asunto());
    }
}
public class AdaptadorSMS implements CanalNotificacion {
    public void enviar(Mensaje m) {
        System.out.println("[SMS] → " + m.destinatario());
    }
}

// Test usa adaptador en memoria
public class AdaptadorFake implements CanalNotificacion {
    public List<Mensaje> enviados = new ArrayList<>();
    public void enviar(Mensaje m) { enviados.add(m); }
}

🛠️ Software que lo implementa

  • Spring Boot + puertos como interfaces
  • NestJS (módulos como adaptadores)
  • Quarkus
  • FastAPI
  • Go con interfaces
  • Clean Architecture de Uncle Bob (variante)

🚫 Cuándo NO usar

  • Para aplicaciones CRUD simples sin lógica de dominio compleja; es sobredimensionado.
  • Equipos pequeños con poco tiempo donde la estructura adicional ralentiza el desarrollo inicial.
  • Cuando todos los casos de uso son simples consultas sin reglas de negocio.
