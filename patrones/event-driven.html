Arquitectura Orientada a Eventos

Diseñar sistemas donde los componentes se comunican produciendo y consumiendo eventos de forma asíncrona, logrando un desacoplamiento extremo.

Arquitectura

📖 Descripción

En la arquitectura orientada a eventos (EDA), los productores emiten eventos cuando cambia su estado; los consumidores suscriben a los eventos de su interés. Nadie espera respuesta inmediata. El broker de eventos (Kafka, RabbitMQ) garantiza la entrega. Esto permite escalar, añadir consumidores y desplegar servicios de forma completamente independiente.

🎯 Casos de uso prácticos

1. Procesamiento de pagos

PagoRealizado → (Inventario actualiza stock, Notificaciones envía email, Analytics registra conversión) — todo en paralelo y desacoplado.

2. Sistema IoT

Sensores emiten eventos de temperatura/humedad; múltiples consumidores procesan alertas, almacenan histórico y actualizan dashboards.

💻 Ejemplos de código

JavaScript
// Event Bus central
class EventBus {
  constructor() { this.handlers = {}; }
  on(evento, handler)  { (this.handlers[evento] = this.handlers[evento] || []).push(handler); }
  emit(evento, payload) {
    console.log(`\n[EventBus] 📢 ${evento}`);
    (this.handlers[evento] || []).forEach(h => {
      // Simula procesamiento asíncrono
      setImmediate(() => h(payload));
    });
  }
}

const bus = new EventBus();

// Productor — Servicio de Pagos
class ServicioPagos {
  constructor(bus) { this.bus = bus; }
  procesarPago(pago) {
    // ... lógica de cobro ...
    console.log(`[Pagos] Pago procesado: #${pago.id}`);
    this.bus.emit('PagoRealizado', { pagoId: pago.id, monto: pago.monto, usuarioId: pago.usuarioId });
  }
}

// Consumidores — desacoplados, no conocen ServicioPagos
bus.on('PagoRealizado', ({ pagoId, monto, usuarioId }) =>
  console.log(`[Inventario] Reduciendo stock por pago #${pagoId}`));

bus.on('PagoRealizado', ({ monto, usuarioId }) =>
  console.log(`[Email] Enviando confirmación a usuario ${usuarioId}: €${monto}`));

bus.on('PagoRealizado', ({ pagoId, monto }) =>
  console.log(`[Analytics] Registrando conversión €${monto} (pago #${pagoId})`));

// Disparar evento
new ServicioPagos(bus).procesarPago({ id: 'PAY-001', monto: 299.99, usuarioId: 'U-42' });
Java
// Evento de dominio
public record PagoRealizado(String pagoId, double monto, String usuarioId) {}

// Productor
@Service
public class ServicioPagos {
    private final ApplicationEventPublisher eventBus;

    public void procesarPago(Pago pago) {
        // ... cobrar ...
        eventBus.publishEvent(new PagoRealizado(pago.getId(), pago.getMonto(), pago.getUsuarioId()));
    }
}

// Consumidores — desacoplados
@Component
public class HandlerInventario {
    @EventListener @Async
    public void on(PagoRealizado e) {
        System.out.println("[Inventario] Reduciendo stock: " + e.pagoId());
    }
}

@Component
public class HandlerNotificaciones {
    @EventListener @Async
    public void on(PagoRealizado e) {
        System.out.println("[Email] Confirmación a " + e.usuarioId());
    }
}

🚫 Cuándo NO usar

  • Cuando necesitas respuesta inmediata y sincrónica del receptor.
  • Para lógicas simples donde la asincronía complica el razonamiento sin beneficio.
  • Sin buena observabilidad (trazas, logs correlacionados), los bugs son muy difíciles de depurar.
  • Cuando la consistencia eventual no es tolerable para el negocio.
