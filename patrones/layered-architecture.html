Arquitectura en Capas

Organizar el código en capas horizontales con responsabilidades bien definidas, donde cada capa solo puede interactuar con la capa inmediatamente inferior.

Arquitectura

📖 Descripción

La arquitectura en capas (N-Tier) divide el sistema en: Presentación (UI/API), Aplicación (casos de uso), Dominio (lógica de negocio) e Infraestructura (BD, servicios externos). Cada capa tiene una responsabilidad única y depende solo de la capa de abajo. Facilita el testing, el reemplazo de implementaciones y la organización del equipo.

🎯 Casos de uso prácticos

1. API REST de e-commerce

Controller → Service → Repository → Database. Cada capa tiene su responsabilidad y puede testearse independientemente.

2. Sistema bancario

Presentación (web/móvil) → Aplicación (transferencias, consultas) → Dominio (reglas contables) → Infraestructura (core bancario).

💻 Ejemplos de código

JavaScript
// Capa de Dominio — entidades y reglas de negocio
class Usuario {
  constructor(id, nombre, email) {
    if (!email.includes('@')) throw new Error('Email inválido');
    this.id = id; this.nombre = nombre; this.email = email;
  }
}

// Capa de Repositorio — acceso a datos
class RepositorioUsuarios {
  constructor() { this.db = new Map(); }
  guardar(usuario)    { this.db.set(usuario.id, usuario); return usuario; }
  buscarPorId(id)     { return this.db.get(id) || null; }
  buscarPorEmail(email) {
    return [...this.db.values()].find(u => u.email === email) || null;
  }
}

// Capa de Servicio (Aplicación) — casos de uso
class ServicioUsuarios {
  constructor(repo) { this.repo = repo; }

  registrar(nombre, email) {
    if (this.repo.buscarPorEmail(email))
      throw new Error('Email ya registrado');
    const usuario = new Usuario(Date.now(), nombre, email);
    return this.repo.guardar(usuario);
  }

  obtener(id) {
    const u = this.repo.buscarPorId(id);
    if (!u) throw new Error('Usuario no encontrado');
    return u;
  }
}

// Capa de Presentación — controlador HTTP
class ControladorUsuarios {
  constructor(servicio) { this.servicio = servicio; }

  registrar(req) {
    try {
      const usuario = this.servicio.registrar(req.nombre, req.email);
      return { status: 201, body: usuario };
    } catch (e) {
      return { status: 400, body: { error: e.message } };
    }
  }
}

// Composición de capas
const repo  = new RepositorioUsuarios();
const svc   = new ServicioUsuarios(repo);
const ctrl  = new ControladorUsuarios(svc);

console.log(ctrl.registrar({ nombre: 'Cesar', email: 'cesar@example.com' }));
console.log(ctrl.registrar({ nombre: 'Cesar', email: 'cesar@example.com' })); // duplicado

// Dominio
public class Usuario {
    private Long id; private String nombre, email;
    public Usuario(Long id, String nombre, String email) {
        if (!email.contains("@")) throw new IllegalArgumentException("Email inválido");
        this.id=id; this.nombre=nombre; this.email=email;
    }
    // getters...
}

// Repositorio (Infraestructura)
@Repository
public interface UsuarioRepository extends JpaRepository<Usuario, Long> {
    Optional<Usuario> findByEmail(String email);
}

// Servicio (Aplicación)
@Service
public class UsuarioService {
    private final UsuarioRepository repo;
    public UsuarioService(UsuarioRepository r) { repo=r; }

    public Usuario registrar(String nombre, String email) {
        if (repo.findByEmail(email).isPresent())
            throw new RuntimeException("Email ya registrado");
        return repo.save(new Usuario(null, nombre, email));
    }
}

// Controlador (Presentación)
@RestController @RequestMapping("/usuarios")
public class UsuarioController {
    private final UsuarioService svc;
    public UsuarioController(UsuarioService s) { svc=s; }

    @PostMapping
    public ResponseEntity<Usuario> registrar(@RequestBody RegistroDto dto) {
        return ResponseEntity.status(201).body(svc.registrar(dto.nombre(), dto.email()));
    }
}

🛠️ Software que lo implementa

  • Spring Boot (Controller/Service/Repository)
  • Django (Views/Forms/Models)
  • ASP.NET Core
  • Ruby on Rails
  • NestJS
  • Laravel

🚫 Cuándo NO usar

  • Cuando las capas se convierten en simples pass-through sin lógica propia (anemic layer anti-pattern).
  • Para microservicios muy pequeños donde la separación en 4 capas es excesiva.
  • Si el rendimiento es crítico; pasar por todas las capas agrega overhead.
