Microservicios

Descomponer una aplicación en servicios pequeños, independientes y desplegables por separado, cada uno responsable de una capacidad de negocio específica.

Arquitectura

📖 Descripción

La arquitectura de microservicios organiza una aplicación como una colección de servicios pequeños que se ejecutan en procesos separados y se comunican mediante APIs bien definidas (REST, gRPC, mensajería). Cada servicio tiene su propia base de datos, puede desplegarse y escalarse independientemente, y pertenece a un equipo autónomo.

🎯 Casos de uso prácticos

1. Plataforma de e-commerce

Servicios separados: Catálogo, Carrito, Pedidos, Pagos, Inventario, Notificaciones — cada uno con su BD y equipo.

2. Plataforma de streaming

Servicios: Autenticación, Catálogo de contenido, Recomendaciones, Reproductor, Facturación — escalados independientemente.

💻 Ejemplos de código

JavaScript
// Simulación de tres microservicios con comunicación HTTP interna
const express = require('express');

// Servicio de Usuarios (puerto 3001)
function crearServicioUsuarios() {
  const app = express();
  app.use(express.json());
  const usuarios = new Map([[1, { id: 1, nombre: 'Cesar', email: 'cesar@example.com' }]]);
  app.get('/usuarios/:id', (req, res) => {
    const u = usuarios.get(parseInt(req.params.id));
    u ? res.json(u) : res.status(404).json({ error: 'No encontrado' });
  });
  return app.listen(3001, () => console.log('UserService :3001'));
}

// Servicio de Pedidos (puerto 3002) — llama a UserService
function crearServicioPedidos() {
  const app = express();
  app.use(express.json());
  app.post('/pedidos', async (req, res) => {
    const { usuarioId, items } = req.body;
    try {
      // Llamada inter-servicio
      const resp = await fetch(`http://localhost:3001/usuarios/${usuarioId}`);
      if (!resp.ok) return res.status(404).json({ error: 'Usuario no encontrado' });
      const usuario = await resp.json();
      const pedido  = { id: Date.now(), usuario: usuario.nombre, items, estado: 'creado' };
      res.status(201).json(pedido);
    } catch (e) {
      res.status(503).json({ error: 'UserService no disponible' });
    }
  });
  return app.listen(3002, () => console.log('OrderService :3002'));
}

crearServicioUsuarios();
crearServicioPedidos();
Java
// Servicio de Pedidos — llama a UserService via REST
@RestController @RequestMapping("/pedidos")
public class PedidoController {
    private final RestTemplate restTemplate;
    private final String userServiceUrl = "http://user-service/usuarios";

    @PostMapping
    public ResponseEntity<Pedido> crearPedido(@RequestBody NuevoPedidoDto dto) {
        // Llamada inter-servicio con circuit breaker (Resilience4j)
        UsuarioDto usuario = restTemplate.getForObject(
            userServiceUrl + "/" + dto.usuarioId(), UsuarioDto.class);
        if (usuario == null)
            return ResponseEntity.notFound().build();
        Pedido pedido = new Pedido(usuario.nombre(), dto.items());
        return ResponseEntity.status(201).body(pedido);
    }
}

  • Consul (service discovery)

🚫 Cuándo NO usar

  • Para startups o MVPs; el overhead operacional (CI/CD, service discovery, observabilidad) es enorme.
  • Cuando el equipo es pequeño y no puede gestionar la complejidad distribuida.
  • Si la latencia de red entre servicios es inaceptable para el caso de uso.
  • Monolito primero (Martin Fowler): empieza monolítico y extrae microservicios cuando los límites sean claros.
