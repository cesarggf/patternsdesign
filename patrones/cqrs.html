CQRS

Separar las operaciones de lectura (Queries) de las de escritura (Commands) en modelos distintos, optimizando cada uno para su propósito.

Arquitectura

📖 Descripción

CQRS (Command Query Responsibility Segregation) divide el modelo de datos en dos: el modelo de escritura procesa comandos que cambian el estado; el modelo de lectura es una proyección optimizada para las consultas. Permiten escalar lecturas y escrituras de forma independiente. Se combina frecuentemente con Event Sourcing.

🎯 Casos de uso prácticos

1. Plataforma de análisis de datos

Los comandos registran eventos de usuario en una BD transaccional; las queries leen de vistas denormalizadas optimizadas para dashboards.

2. Sistema de inventario de alto tráfico

Las escrituras (ajustar stock) van a la BD principal; las lecturas (catálogo) van a un caché Redis actualizado por proyecciones.

💻 Ejemplos de código

JavaScript
// ── LADO ESCRITURA (Commands) ──
class ComandoCrearPedido {
  constructor(usuarioId, items) { this.usuarioId = usuarioId; this.items = items; }
}

class ManejadorCrearPedido {
  constructor(repo, eventBus) { this.repo = repo; this.eventBus = eventBus; }

  manejar(cmd) {
    const pedido = { id: Date.now(), usuarioId: cmd.usuarioId, items: cmd.items, estado: 'creado' };
    this.repo.guardar(pedido);
    this.eventBus.publicar('PedidoCreado', pedido); // actualiza modelo lectura
    return pedido.id;
  }
}

// ── LADO LECTURA (Queries) ──
class PedidosReadModel {
  constructor() { this.proyecciones = new Map(); }

  // Se actualiza al recibir eventos del lado escritura
  aplicar(evento, datos) {
    if (evento === 'PedidoCreado') {
      this.proyecciones.set(datos.id, {
        id: datos.id,
        resumen: `Pedido de ${datos.items.length} item(s)`,
        estado: datos.estado
      });
    }
  }

  obtenerPedido(id)   { return this.proyecciones.get(id); }
  listarPedidos()     { return [...this.proyecciones.values()]; }
}

// Bus de eventos (conecta escritura → lectura)
const eventBus = { listeners: {}, publicar(ev, d) { (this.listeners[ev]||[]).forEach(f=>f(d)); }, on(ev,f){ (this.listeners[ev]=this.listeners[ev]||[]).push(f); } };

const readModel = new PedidosReadModel();
eventBus.on('PedidoCreado', datos => readModel.aplicar('PedidoCreado', datos));

const manejador = new ManejadorCrearPedido({ guardar: ()=>{} }, eventBus);
const id = manejador.manejar(new ComandoCrearPedido('u1', ['Laptop', 'Mouse']));

console.log(readModel.obtenerPedido(id));
console.log(readModel.listarPedidos());
Java
// Comando
public record CrearPedidoCmd(String usuarioId, List<String> items) {}

// Handler de escritura
@Service
public class CrearPedidoHandler {
    private final PedidoRepository writeRepo;
    private final ApplicationEventPublisher eventBus;

    public Long handle(CrearPedidoCmd cmd) {
        Pedido pedido = new Pedido(cmd.usuarioId(), cmd.items());
        writeRepo.save(pedido);
        eventBus.publishEvent(new PedidoCreadoEvent(pedido));
        return pedido.getId();
    }
}

// Read Model (proyección)
@Component
public class PedidosReadModel {
    private Map<Long, PedidoResumenDto> cache = new ConcurrentHashMap<>();

    @EventListener
    public void on(PedidoCreadoEvent e) {
        cache.put(e.pedido().getId(),
            new PedidoResumenDto(e.pedido().getId(), e.pedido().getItems().size(), "creado"));
    }
    public PedidoResumenDto obtener(Long id) { return cache.get(id); }
}

🚫 Cuándo NO usar

  • Para aplicaciones CRUD simples sin diferencia de carga entre lecturas y escrituras.
  • Si el equipo no está familiarizado con el patrón; la curva de aprendizaje es alta.
  • Cuando la consistencia eventual entre modelos de lectura y escritura no es aceptable para el negocio.
