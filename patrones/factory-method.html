Factory Method

Definir una interfaz para crear objetos, pero dejar que las subclases decidan qué clase instanciar.

Creacional

📖 Descripción

Factory Method delega la responsabilidad de crear objetos a subclases, eliminando el acoplamiento directo entre el creador y los productos concretos. El creador declara un método de fábrica abstracto, y cada subclase lo implementa para producir el tipo de objeto que necesita.

🎯 Casos de uso prácticos

1. Generador de notificaciones

Un sistema envía notificaciones por email, SMS o push. Cada canal tiene su propia fábrica.

2. Parsers de documentos

Una aplicación lee XML, JSON o CSV. Cada formato tiene su propio parser creado por su fábrica.

💻 Ejemplos de código

JavaScript
// Producto abstracto
class Notificacion {
  enviar(mensaje) { throw new Error('Método abstracto'); }
}

// Productos concretos
class NotificacionEmail extends Notificacion {
  enviar(mensaje) { console.log(`📧 Email: ${mensaje}`); }
}
class NotificacionSMS extends Notificacion {
  enviar(mensaje) { console.log(`📱 SMS: ${mensaje}`); }
}

// Creador con factory method
class CreadorNotificacion {
  crearNotificacion() { throw new Error('Método abstracto'); }
  notificar(mensaje) { this.crearNotificacion().enviar(mensaje); }
}

class CreadorEmail extends CreadorNotificacion {
  crearNotificacion() { return new NotificacionEmail(); }
}
class CreadorSMS extends CreadorNotificacion {
  crearNotificacion() { return new NotificacionSMS(); }
}

// Uso
const creadores = [new CreadorEmail(), new CreadorSMS()];
creadores.forEach(c => c.notificar('¡Pedido confirmado!'));
Java
// Producto abstracto
public interface Notificacion {
    void enviar(String mensaje);
}

// Productos concretos
public class NotificacionEmail implements Notificacion {
    public void enviar(String mensaje) {
        System.out.println("📧 Email: " + mensaje);
    }
}
public class NotificacionSMS implements Notificacion {
    public void enviar(String mensaje) {
        System.out.println("📱 SMS: " + mensaje);
    }
}

// Creador abstracto
public abstract class CreadorNotificacion {
    public abstract Notificacion crearNotificacion();

    public void notificar(String mensaje) {
        crearNotificacion().enviar(mensaje);
    }
}

// Creadores concretos
public class CreadorEmail extends CreadorNotificacion {
    public Notificacion crearNotificacion() { return new NotificacionEmail(); }
}
public class CreadorSMS extends CreadorNotificacion {
    public Notificacion crearNotificacion() { return new NotificacionSMS(); }
}

🚫 Cuándo NO usar

  • Cuando solo tienes un tipo de producto y no se espera variación; añade complejidad innecesaria.
  • Si la jerarquía de clases ya es muy profunda, añadir más subclases empeora la mantenibilidad.
  • Para lógica de creación trivial, un simple constructor o función es suficiente.
