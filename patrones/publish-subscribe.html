Publish-Subscribe

Permitir que múltiples consumidores reciban cada mensaje publicado en un canal, sin que el emisor conozca a los receptores.

Integración

📖 Descripción

En Publish-Subscribe el publicador envía mensajes a un topic o canal de difusión. Todos los suscriptores activos reciben una copia del mensaje. Difiere del Message Channel point-to-point en que el mensaje llega a todos los interesados. Es la base de la arquitectura orientada a eventos y los sistemas reactivos.

🎯 Casos de uso prácticos

1. Actualización de caché distribuida

Cuando cambia un precio en la BD, se publica un evento; todos los nodos que tienen ese precio en caché lo invalidan.

2. Notificaciones en tiempo real

Un servicio de chat publica mensajes en un topic; todas las sesiones WebSocket suscritas al canal reciben la actualización.

💻 Ejemplos de código

JavaScript
class PubSub {
  constructor() { this.topics = {}; }

  suscribir(topic, fn) {
    if (!this.topics[topic]) this.topics[topic] = [];
    this.topics[topic].push(fn);
    return () => this.topics[topic] = this.topics[topic].filter(f => f !== fn);
  }

  publicar(topic, datos) {
    console.log(`[PubSub] topic='${topic}' → ${this.topics[topic]?.length ?? 0} suscriptores`);
    (this.topics[topic] || []).forEach(fn => fn(datos));
  }
}

const bus = new PubSub();

// Suscriptores al topic 'precio:actualizado'
const cancelarA = bus.suscribir('precio:actualizado', ({ sku, precio }) =>
  console.log(`[CachéNodo1] Invalidando ${sku}`));
bus.suscribir('precio:actualizado', ({ sku, precio }) =>
  console.log(`[CachéNodo2] Precio nuevo ${sku}: €${precio}`));
bus.suscribir('precio:actualizado', ({ sku, precio }) =>
  console.log(`[Analytics] Registrando cambio de precio ${sku}`));

// Publicar evento
bus.publicar('precio:actualizado', { sku: 'LAPTOP-X1', precio: 899 });

// Cancelar suscripción
cancelarA();
bus.publicar('precio:actualizado', { sku: 'LAPTOP-X1', precio: 849 });
Java
import java.util.*;
import java.util.function.*;

public class PubSub<T> {
    private Map<String, List<Consumer<T>>> topics = new HashMap<>();

    public void suscribir(String topic, Consumer<T> fn) {
        topics.computeIfAbsent(topic, k -> new ArrayList<>()).add(fn);
    }
    public void publicar(String topic, T datos) {
        System.out.println("[PubSub] topic='" + topic + "'");
        topics.getOrDefault(topic, List.of()).forEach(fn -> fn.accept(datos));
    }

    public static void main(String[] args) {
        PubSub<Map<String,Object>> bus = new PubSub<>();
        bus.suscribir("precio:actualizado",
            m -> System.out.println("[Caché] Invalidando " + m.get("sku")));
        bus.suscribir("precio:actualizado",
            m -> System.out.println("[Analytics] Cambio de precio: " + m));
        bus.publicar("precio:actualizado",
            Map.of("sku","LAPTOP-X1","precio",899));
    }
}

Notificaciones]

🛠️ Software que lo implementa

  • Apache Kafka
  • Redis Pub/Sub
  • AWS SNS
  • Google Cloud Pub/Sub
  • RabbitMQ (fanout exchange)
  • NATS

🚫 Cuándo NO usar

  • Cuando solo hay un consumidor; un canal point-to-point es suficiente y más simple.
  • Si el orden de entrega entre suscriptores es crítico y el broker no lo garantiza.
  • Para operaciones que deben ejecutarse exactamente una vez; Pub/Sub puede causar procesamientos duplicados sin idempotencia.
